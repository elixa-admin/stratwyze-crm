GitHub: Version Control & CI/CD

GitHub is where all AISP code lives. We use a centralized organization with standardized repo names and branch strategies.

Organization Setup: AISP-Company

URL: https://github.com/AISP-Company

Organization Structure:
  • Single org: AISP-Company
  • Billing: Centralized
  • Teams: Engineering, Product, Sales, Finance (by director)
  • Visibility: Private repos (unless publicly documenting standards)

Repository Naming Convention

Type Name Format Example Visibility
Standards & Policy aisp-standards aisp-standards Private or Public
Main Product aisp-portal aisp-portal Private
Project aisp-project-[name] aisp-project-alpha Private

Branch Strategy (All Repos)

🔗 Note: All projects follow the same branch strategy. This ensures consistency across the entire AISP organization.
Branch Purpose Deployment Protection
main Production code (tagged releases) Vercel Production Require PR review, tests pass
dev Development (Sprints 1-9 accumulate here) Vercel Staging Require PR review
feature/* Feature work (branch per feature) Vercel Preview (auto) None (user repo)
hotfix/* Production emergency fixes Manual (to main) Require review

Workflow: Sprint to Production

Sprints 1-9 (Development):
  1. Create feature branch from dev: git checkout -b feature/your-feature dev
  2. Work on feature (commit daily)
  3. Open PR against dev
  4. Code review, tests pass
  5. Merge to dev
  6. Vercel auto-deploys to staging
Sprint 10 (Deployment):
  1. Final integration testing on dev
  2. Create PR: dev → main
  3. Code review, verify all tests pass
  4. Merge to main
  5. Tag release: git tag v1.2 main && git push origin v1.2
  6. Vercel auto-deploys to production

GitHub Actions / CI Workflow

Every repo has a .github/workflows/ folder with:

  • Unit tests on every PR
  • Integration tests on PR to dev
  • E2E tests on PR to main
  • Build verification (no size regressions)
  • Linting & formatting checks

Setup: First Time

# Clone the repo git clone https://github.com/AISP-Company/aisp-project-alpha.git cd aisp-project-alpha # Create development branch if needed git checkout -b dev origin/dev # Create your feature branch git checkout -b feature/your-feature dev # Make changes, commit, push git add . git commit -m "feat: your feature description" git push -u origin feature/your-feature # Open PR in GitHub UI # → Compare: dev ... feature/your-feature # → Request reviews # → Wait for CI to pass # → Merge once approved

Vercel: Deployment & Environments

Vercel handles all deployments. Every project has production and staging environments. Previews auto-generate per PR.

Project Setup: Production + Staging

Each project gets TWO Vercel projects:

Environment Name URL Branch Deploy Trigger
Production aisp-project-alpha project-alpha.vercel.app main Merge to main (Sprint 10)
Staging aisp-project-alpha-staging project-alpha-staging.vercel.app dev Merge to dev (Sprints 1-9)
Preview Auto-created auto-preview.vercel.app feature/* Every PR (auto)

Environment Variables Setup

Configure in Vercel Dashboard → Settings → Environment Variables

Production Variables (main branch):
  • API_URL = production API endpoint
  • DATABASE_URL = production database (secret)
  • LOG_LEVEL = "error" (minimal logging)
  • Any other production secrets
Staging Variables (dev branch):
  • API_URL = staging API endpoint
  • DATABASE_URL = staging database (secret)
  • LOG_LEVEL = "debug" (detailed logging)
  • Same keys as production, different values
Preview Variables (feature/* branches):
  • Uses staging variables by default
  • Override for testing (optional)

Deployment Workflow

Automatic Deployments:

  1. Push code to GitHub
  2. GitHub notifies Vercel
  3. Vercel builds (2-5 min)
  4. Vercel deploys (auto)
  5. Preview/Staging/Production live

Rollback Procedure (Emergency)

⚠️ Use only if production is broken. For normal fixes, just push a new commit.
  1. GitHub: Identify the good commit (e.g., v1.1 tag)
  2. Git: git checkout v1.1 && git push -f origin main (force push old version)
  3. Vercel: Revert automatically triggers new deployment
  4. Verify: Check https://project-alpha.vercel.app is back online
  5. Post-mortem: Figure out what broke in the bad commit

Monitoring & Logs

Vercel Dashboard → Deployments → [Environment] → Logs:

  • Build logs: See compile errors, dependencies
  • Runtime logs: See app errors (production)
  • Function logs: Serverless function execution

Linear tracks all work: features, bugs, refactors. Integrates with GitHub for automatic PR closing.

Workspace Setup

URL: https://linear.app/aisp-company

Team Setup:
  • Single workspace: AISP-Company
  • Projects: One per Wave/Initiative
  • Cycles: 2-week sprints (matching AISP sprints)
  • Members: All directors + engineers

Project Structure

Project Scope Owner Cycle
PORTAL AISP Portal main product Anderson (CTO) 2-week sprints
ALPHA Project Alpha Project lead 2-week sprints
INFRA Infrastructure & DevOps Anderson (CTO) 2-week sprints

Workflow States

Backlog → Todo → In Progress → In Review → Done Backlog: Not started, not committed to sprint Todo: In current sprint, not started In Progress: Engineer working on it In Review: PR open, waiting for review Done: PR merged, shipped

Creating & Managing Issues

When creating an issue:
  • Clear title: "Add OAuth2 support for Google"
  • Description: What + Why + Success criteria
  • Assign to: The engineer who'll do it
  • Priority: P1 (urgent) → P4 (nice-to-have)
  • Project: PORTAL, ALPHA, or INFRA
  • Cycle: Current sprint (or backlog if future)

GitHub Integration: Auto-Close

When you include an issue number in your GitHub commit, Linear auto-closes the issue:

git commit -m "feat: add OAuth2 Google support (PORTAL-42)" # ↑ # Closes PORTAL-42 in Linear

Linear auto-closes when:

  • PR is merged to main or dev
  • Commit message includes issue number (e.g., PORTAL-42, ALPHA-18)
  • Issue moves to Done, date set to merge date

Pattern Library: Knowledge Base & Smart Toolkit

The Pattern Library is a shared SQLite database of bug fixes, solutions, and learned patterns used across all AISP projects.

What It Stores

  • Bug fixes: "How to fix auth token expiry"
  • Solutions: "How to optimize slow database queries"
  • Patterns: "Recurring infrastructure issues"
  • Skills: Automated fixes that run on detection
  • Confidence scores: How reliable is each pattern?

Location & Access

Local: ~/.claude/knowledge_base/patterns.db GitHub: AISP-Company/aisp-standards (synchronized weekly) Backup: AWS S3 (automated daily)

How to Query Patterns

Example: Find all auth-related fixes

sqlite3 ~/.claude/knowledge_base/patterns.db sqlite> SELECT id, name, tags, confidence_score FROM patterns WHERE tags LIKE '%auth%' AND confidence_score > 0.85 ORDER BY confidence_score DESC;

Example: Find patterns by confidence

sqlite> SELECT id, name, usage_count, success_rate FROM patterns WHERE confidence_score >= 0.90 AND usage_count > 3 ORDER BY success_rate DESC;

How to Add a New Pattern

After fixing a bug:
  1. Document the fix: symptoms, root cause, solution
  2. Add to pattern database: INSERT INTO patterns ...
  3. Tag it: auth, database, performance, etc.
  4. Set confidence: How sure are you this will work again?
  5. Commit to git (weekly sync to GitHub)
INSERT INTO patterns ( id, name, tags, symptoms, root_cause, solution, confidence_score, usage_count, last_used ) VALUES ( 'auth_token_expiry', 'Auth Token Expiry Fix', 'auth,token,401,api', 'Token expired,401 Unauthorized', 'OAuth token expired; refresh failed', 'Run refresh token endpoint; cache new token', 0.98, 5, '2026-06-20' );

Weekly Sync: SQLite ↔ GitHub

Automatic (every Sunday 11 PM):

  1. Export patterns.db as JSON
  2. Commit to AISP-Company/aisp-standards
  3. All projects pull latest patterns (auto on startup)

Manual (if urgent):

bash ~/.claude/sync_patterns_to_github.sh

Cross-Project Sharing

All AISP projects share ONE pattern library.

  • Pattern from Project Alpha is available to AISP Portal
  • No duplication: if both had same bug, pattern found once
  • Learning compounds: more projects = smarter patterns
  • Tags help discovery: search by domain, not project

Confidence Scoring (Auto-Update)

Confidence Meaning Auto-Run? Example
0.90+ Highly reliable Yes (aggressive) auth_token_refresh (used 10x, 100% success)
0.85-0.89 Reliable Yes (with notice) db_query_optimize (used 5x, 95% success)
0.70-0.84 Suggest only No (human picks) new pattern, or 1-2 uses
<0.70 Archive No Known to be unreliable; keep for history
